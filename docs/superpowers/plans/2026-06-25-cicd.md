# CI/CD com GitHub Actions — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Configurar dois pipelines GitHub Actions independentes (um para `backend/`, outro para `frontend/`) que rodam lint + testes unitários, publicam imagens Docker no GHCR e fazem deploy via SSH em uma VPS Hostinger.

**Architecture:** Dois arquivos de workflow em `.github/workflows/` com path filtering por serviço. Cada workflow tem três jobs em cadeia: `ci → docker → deploy`. As imagens são publicadas no GitHub Container Registry e a VPS consome essas imagens via `docker-compose.prod.yml`.

**Tech Stack:** GitHub Actions, GHCR, Docker Buildx, `appleboy/ssh-action`, Docker Compose.

## Global Constraints

- Runner: `ubuntu-latest` em todos os jobs
- Node: 22
- Imagens no GHCR: `ghcr.io/jumaelmartins/portfolio-manager-{backend,frontend}`
- Tags de imagem: `:latest` e `:sha-{github.sha}`
- VPS working directory: `/opt/portfolio-manager`
- Compose file de produção: `docker-compose.prod.yml`
- Secrets do repositório: `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`, `CR_PAT`

## Task 1: Criar `docker-compose.prod.yml`

O `docker-compose.yml` existente usa `build:` contexts — constrói as imagens localmente. Na VPS, o compose precisa referenciar as imagens do GHCR para que `docker compose pull` funcione.

**Files:**
- Create: `docker-compose.prod.yml` (raiz do repositório)

- [ ] **Step 1: Criar o arquivo**

```yaml
# docker-compose.prod.yml
name: portfolio-manager

services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: portfolio
      POSTGRES_PASSWORD: portfolio_pass
      POSTGRES_DB: portfolio_db
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U portfolio -d portfolio_db"]
      interval: 5s
      timeout: 5s
      retries: 10
    restart: unless-stopped

  backend:
    image: ghcr.io/jumaelmartins/portfolio-manager-backend:latest
    ports:
      - "3000:3000"
    volumes:
      - uploads:/app/uploads
    environment:
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: postgresql://portfolio:portfolio_pass@db:5432/portfolio_db
      JWT_SECRET: ${JWT_SECRET}
      JWT_EXPIRATION: ${JWT_EXPIRATION:-24h}
      FRONTEND_URL: ${FRONTEND_URL}
      CORS_ORIGINS: ${CORS_ORIGINS}
      BACKEND_PUBLIC_URL: ${BACKEND_PUBLIC_URL}
      EMAIL_TRANSPORT: ${EMAIL_TRANSPORT:-smtp}
      EMAIL_FROM: ${EMAIL_FROM}
      GOOGLE_CLIENT_ID: ${GOOGLE_CLIENT_ID:-disabled}
      GOOGLE_CLIENT_SECRET: ${GOOGLE_CLIENT_SECRET:-disabled}
      GOOGLE_CALLBACK_URL: ${GOOGLE_CALLBACK_URL}
    depends_on:
      db:
        condition: service_healthy
    healthcheck:
      test:
        ["CMD-SHELL", "wget --quiet --output-document=- http://localhost:3000 >/dev/null 2>&1"]
      interval: 5s
      timeout: 5s
      retries: 20
      start_period: 15s
    restart: unless-stopped

  frontend:
    image: ghcr.io/jumaelmartins/portfolio-manager-frontend:latest
    ports:
      - "3001:3001"
    environment:
      BACKEND_URL: http://backend:3000
      NEXT_PUBLIC_APP_URL: ${NEXT_PUBLIC_APP_URL}
      SESSION_COOKIE_NAME: pm_session
      VERIFICATION_COOKIE_NAME: pm_verification
      OAUTH_STATE_COOKIE_NAME: pm_oauth_state
    depends_on:
      backend:
        condition: service_healthy
    restart: unless-stopped

volumes:
  pgdata:
  uploads:
```

As variáveis sem default (ex.: `${JWT_SECRET}`) devem ser definidas em um arquivo `.env` dentro de `/opt/portfolio-manager/` na VPS. O Compose carrega `.env` automaticamente.

## Task 2: Criar `.github/workflows/backend.yml`

Pipeline completo do backend: lint + testes unitários → build Docker → deploy via SSH.

**Files:**
- Create: `.github/workflows/backend.yml`

- [ ] **Step 1: Criar o diretório e o arquivo**

```bash
mkdir -p .github/workflows
```

Conteúdo de `.github/workflows/backend.yml`:

```yaml
name: Backend CI/CD

on:
  push:
    branches: [master]
    paths:
      - "backend/**"
      - ".github/workflows/backend.yml"

jobs:
  ci:
    name: Lint & Test
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: backend
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: backend/package-lock.json

      - run: npm ci
      - run: npm run lint
      - run: npm test

  docker:
    name: Build & Push
    needs: ci
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/build-push-action@v6
        with:
          context: ./backend
          push: true
          cache-from: type=gha
          cache-to: type=gha,mode=max
          tags: |
            ghcr.io/jumaelmartins/portfolio-manager-backend:latest
            ghcr.io/jumaelmartins/portfolio-manager-backend:sha-${{ github.sha }}

  deploy:
    name: Deploy
    needs: docker
    runs-on: ubuntu-latest
    steps:
      - uses: appleboy/ssh-action@v1
        env:
          CR_PAT: ${{ secrets.CR_PAT }}
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          envs: CR_PAT
          script: |
            echo "$CR_PAT" | docker login ghcr.io -u jumaelmartins --password-stdin
            cd /opt/portfolio-manager
            docker compose -f docker-compose.prod.yml pull backend
            docker compose -f docker-compose.prod.yml up -d backend
```

## Task 4: Configurar secrets e preparar a VPS

Sem código — checklist de configuração manual que desbloqueia o job `deploy`.

**Part A — Secrets no GitHub**

Acessar: `https://github.com/jumaelmartins/portfolio-manager/settings/secrets/actions`

Criar os seguintes secrets (New repository secret):

| Nome | Valor |
|---|---|
| `VPS_HOST` | IP público da VPS (ex: `123.45.67.89`) |
| `VPS_USER` | Usuário SSH (ex: `root`) |
| `VPS_SSH_KEY` | Conteúdo da chave privada SSH (o arquivo inteiro, incluindo `-----BEGIN...`) |
| `CR_PAT` | GitHub PAT com escopo `read:packages` (gerado em Settings → Developer settings → Personal access tokens → Tokens classic) |

**Part B — Preparar a VPS**

Conectar via SSH e executar:

```bash
# Instalar Docker (se ainda não instalado)
curl -fsSL https://get.docker.com | sh

# Criar diretório de trabalho
mkdir -p /opt/portfolio-manager

# Criar arquivo .env com as variáveis de produção
cat > /opt/portfolio-manager/.env << 'EOF'
JWT_SECRET=<segredo-forte-aqui>
JWT_EXPIRATION=24h
FRONTEND_URL=https://<seu-dominio>
CORS_ORIGINS=https://<seu-dominio>
BACKEND_PUBLIC_URL=https://<seu-dominio>
EMAIL_TRANSPORT=smtp
EMAIL_FROM=Portfolio Manager <noreply@<seu-dominio>>
GOOGLE_CLIENT_ID=<client-id>
GOOGLE_CLIENT_SECRET=<client-secret>
GOOGLE_CALLBACK_URL=https://<seu-dominio>/auth/google/callback
NEXT_PUBLIC_APP_URL=https://<seu-dominio>
EOF
```

**Part C — Copiar `docker-compose.prod.yml` para a VPS**

Executar na sua máquina local:

```bash
scp docker-compose.prod.yml <VPS_USER>@<VPS_HOST>:/opt/portfolio-manager/docker-compose.prod.yml
```

**Part D — Primeiro deploy manual para validar**

Na VPS, executar o primeiro deploy manualmente para garantir que tudo funciona antes de depender do pipeline:

```bash
cd /opt/portfolio-manager

# Login no GHCR com o CR_PAT
echo "<CR_PAT>" | docker login ghcr.io -u jumaelmartins --password-stdin

# Subir tudo pela primeira vez (banco + backend + frontend)
docker compose -f docker-compose.prod.yml up -d
```

Verificar que os três serviços estão rodando:
```bash
docker compose -f docker-compose.prod.yml ps
```

Expected: `db`, `backend` e `frontend` com status `healthy` ou `running`.

**Part E — Re-executar os workflows com retry**

Após configurar os secrets, abrir o GitHub Actions, escolher a última execução do backend ou frontend com o job `deploy` com falha, e clicar em **Re-run failed jobs**.

O job `deploy` deve passar desta vez.
